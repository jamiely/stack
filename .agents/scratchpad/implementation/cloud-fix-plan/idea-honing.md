# Idea Honing

## 2026-03-31
- Q1: What is explicitly out of scope for this cloud fix milestone so we avoid overreaching beyond the plan (e.g., weather systems, parallax layers, or non-cloud background refactors)?
- A1: Out of scope for this milestone is anything beyond the cloud subsystem rewrite and its direct integration. Specifically excluded: (1) weather features (rain/snow/wind systems, time-of-day/lighting weather coupling), (2) adding new background/parallax layer systems unrelated to clouds, (3) broad non-cloud background rendering refactors, (4) camera-system redesign beyond consuming existing camera/world bounds needed for cloud lifecycle math, (5) unrelated gameplay/mechanics tuning, and (6) net-new performance-mode architecture (e.g., dedicated low-end profile), aside from maintaining no noticeable regression. In scope remains only deterministic cloud simulation + recycle policy, front/back lane behavior, rounded-lobe cloud visuals, cloud debug controls/clamps, cloud test-surface expansion, and required cloud-focused tests/docs updates.
- Q2: What are the concrete acceptance criteria for this milestone at ship time—specifically, which deterministic behavior checks, visual outcomes, and performance/regression thresholds must pass for cloud changes to be considered complete?
- A2: Ship-time acceptance for this milestone is: (1) **Deterministic behavior checks** — all five cloud acceptance checks from the plan must pass in automation: camera-ascent screen-Y progression, recycle-only-after-threshold, spawn-X visible-width sampling with edge clipping allowance, repeatable trajectories for same seed+steps, and maintained front/back lane mix; deterministic replay assertions should compare per-cloud diagnostics from test API snapshots and match exactly within floating tolerance (≤1e-4). (2) **Visual outcomes** — cloud layer remains visibly populated before and after scripted ascent (no random hard disappearance), cloud nodes render using the new rounded-lobe procedural style (smooth/non-pixelized), and front/behind layering is visually and state-wise represented (at least one cloud in each lane during tuned mixed-lane scenarios). (3) **Runtime/debug surface** — `?debug` exposes the cloud controls (count/density, horizontal drift speed, spawn-above band, despawn-below threshold; plus lane controls if added), clamps enforce valid ranges, and live slider updates alter observed cloud behavior without restart. (4) **Performance/regression thresholds** — no regression in existing core suites: unit tests pass, Playwright core gameplay flows pass, and cloud changes do not break current performance smoke expectations (including responsive input and bounded active-object behavior under low-quality archival scenario). (5) **Documentation/testability completion** — test-mode `getState()` includes cloud diagnostics required for deterministic assertions, and docs (`docs/features.md`, README if surfaced behavior changed) are updated to match the shipped cloud behavior and controls.
- Q3: For edge-case runtime tuning values, what behavior is required as “correct” when cloud controls hit extremes (e.g., count/density at minimum or maximum, zero horizontal drift, or spawn/despawn thresholds configured too tightly) so implementation and tests can distinguish intentional sparse/static clouds from bugs?
- A3: Treat extreme debug settings as valid, deterministic operating modes, not implicit error states. Required semantics: (1) **Cloud count/density minimum**: permit intentionally sparse output (including exactly one cloud if min clamp is 1), but never an accidental empty layer while cloud channel is enabled and count > 0; if count is clamped to 0 in future, expected result is a cleanly empty cloud layer with no recycle churn. (2) **Cloud count/density maximum**: system must honor the clamped max without overspawning, preserve deterministic ordering/IDs, and keep lifecycle stable (no duplicate IDs or runaway creation). (3) **Horizontal drift = 0**: X position is stationary except for recycle re-seeding; this is a correct “static sky” mode and should not fail motion assertions that specifically target Y/screen movement from camera ascent. (4) **Spawn/despawn bands tightened or inverted by input**: clamp/sanitize to a valid effective configuration before simulation (e.g., enforce non-negative bands and a minimum separation between spawn-above and despawn-below semantics) so clouds do not thrash-recycle every frame; once sanitized, recycle should still occur only after crossing effective despawn threshold. (5) **Very tight but valid thresholds**: rapid recycle cadence is acceptable if deterministic and threshold-driven, but must still satisfy “no pre-threshold disappearance” in diagnostics. (6) **Test expectations at extremes**: tests should assert policy compliance (clamp result, deterministic state progression, threshold-gated recycle, and lane-mix preservation when count allows) rather than assuming visually dense or drifting clouds.

- Requirements checkpoint: core functional scope, measurable acceptance gates, and edge-case semantics are now explicit and sufficient to proceed to detailed design/implementation planning without additional blocking requirement questions.
