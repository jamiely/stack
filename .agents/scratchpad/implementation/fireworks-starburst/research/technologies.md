# Technologies & Tooling — Fireworks Starburst

## Runtime stack
- **TypeScript + Vite + Three.js** power the game runtime.
- Fireworks visuals are DOM-overlay actor nodes generated by `Game` while simulation runs in pure logic modules.

**Evidence**

## Practical constraints discovered
1. **No current screenshot assertion pattern in e2e tests**
   - Existing tests use state/DOM assertions and ad-hoc attachments, but not `expect(...).toHaveScreenshot(...)` baselines.
   - Implication: Step 6 introduces new baseline snapshot infra for this suite.
   - Evidence: fireworks/clouds specs currently use evaluate/assertions and manual screenshots (`tests/e2e/fireworks.spec.ts:3-374`, `tests/e2e/clouds.spec.ts:96-124`).

2. **Canonical screenshot contract is defined in design artifacts, not yet in tests**
   - Required baseline path/tolerances/capture anchor are specified in requirements/design docs.
   - Implication: Builder should implement that contract exactly in `tests/e2e/fireworks.spec.ts`.
   - Evidence: contract details (`.agents/scratchpad/implementation/fireworks-starburst/requirements.md:32-41`, `.agents/scratchpad/implementation/fireworks-starburst/design.md:32-36`).
